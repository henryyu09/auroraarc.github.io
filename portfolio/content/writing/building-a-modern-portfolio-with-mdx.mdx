## Why MDX?

For a portfolio that showcases both projects and writing, I wanted content that was more expressive than plain Markdown but didn't require a CMS. MDX — Markdown with JSX — lets me embed interactive components (charts, diagrams, code blocks with syntax highlighting) directly into my writing.

The tradeoff: MDX requires compilation at build time, which means content changes require a rebuild. For a portfolio that I update infrequently, that's acceptable.

## The Stack

- **Next.js 15** with App Router for routing and RSC
- **next-mdx-remote** for compiling MDX at build time
- **Tailwind CSS v4** for styling (the new CSS-first config is much cleaner)
- **Chart.js + react-chartjs-2** for result charts
- **Mermaid** for diagrams
- **Inter, JetBrains Mono, Sora** for typography

## Design System Choices

I built a CSS custom property token system before writing any components. This might feel like overkill for a small site, but it made two things much easier:

1. **Dark mode**: Every color references a CSS variable. Dark mode is a single class toggle on `<html>`.
2. **Consistency**: Component spacing, type scale, and border radius all come from tokens, not arbitrary values.

The muted green accent (`#3A7D5C`) was chosen deliberately. It's technical without being sterile, and it works well in both light and dark themes.

## What I'd Do Differently

- **More test coverage earlier**: I added Vitest and Playwright after the fact. Next time I'd write tests alongside components.
- **Content validation**: MDX frontmatter is parsed manually. A schema validator would catch typos earlier.
- **Image optimization**: For project screenshots, next/image with a CDN would be better than static imports.
